# prototype-harness-extend

> **Category**: UI / Prototyping
> **Last Reviewed**: 2026-06-18
> **Exemplar rationale**: Factory files are entity-specific (one per `sprk_*` table); no canonical artifact. The 6-step generation workflow is the contract.

---

## Purpose

Scaffold a new entity factory + register it in a seed preset so harnesses can render UIs that query the entity. Companion to `prototype-harness-setup` — that skill creates the harness shell; this one populates the data the harness's UI reads.

Uses Dataverse MCP (when available) to fetch the live schema so the factory matches the real entity contract. Falls back to user-supplied field list when MCP is unavailable or the entity is unpublished.

**Reduces a 30-minute manual derivation** (open Dataverse maker → list columns → infer types → hand-write defaults → register in seed index) **to ~3 minutes**.

**Sandbox safety**: Seed data is in-memory only, lives in `c:/code_files/spaarke-prototype/_infra/seed/`. Production code never imports it. See `prototype-harness-setup` "Sandbox Safety" section.

---

## Applies When

- An existing harness needs to query an entity that has no factory in `_infra/seed/factories/`
- A new project's harness setup is gated on missing seed data
- A regarding-picker or lookup widget shows empty because the picker target entity isn't seeded
- **Trigger phrases**: `/prototype-harness-extend`, "add entity to prototype seed", "create factory for sprk_X", "extend the harness with X entity", "seed data for sprk_X"

**Do NOT use** when:
- The entity already has a factory (extend the existing factory or override at mount time instead — see PROTOTYPE-UI-SYSTEM-GUIDE.md "Seed data — How to override seed data at mount time")
- The user needs a one-off scenario (empty state, single high-priority record) — override at `installMocks({ seed: { records: {...} } })` directly in `main.tsx`, no factory needed

## Workflow

### Step 1: Gather Inputs

**Entity logical name** (required):
```
ASK: "Entity logical name? (e.g., 'sprk_matter', 'sprk_event')"
```

**Schema source** (auto-detect):
```
IF Dataverse MCP is available:
  USE mcp__dataverse__describe to fetch entity metadata
  → list columns, types, lookup targets, option-set values
ELSE:
  ASK: "Dataverse MCP unavailable. Provide field list (comma-separated 'field:type'):
        Example: sprk_name:string, sprk_status:optionset, sprk_amount:money, sprk_clientid:lookup(account)"
```

**Preset(s) to extend** (optional):
```
IF a specific harness consumer needs this entity:
  ASK: "Add to which seed preset? (existing preset name, or 'new' to scaffold one)"
ELSE:
  Skip preset extension — just produce the factory; consumer can register later
```

**Record count** (default 10):
```
ASK only if not obvious from context: "How many records? (default 10)"
```

---

### Step 2: Schema Discovery via Dataverse MCP

**WHEN MCP is available**, run:

```typescript
mcp__dataverse__describe({ entityName: "sprk_<entity>" })
```

Expected response shape:
```json
{
  "logicalName": "sprk_matter",
  "primaryIdAttribute": "sprk_matterid",
  "primaryNameAttribute": "sprk_name",
  "attributes": [
    { "logicalName": "sprk_name", "type": "String", "maxLength": 100, "required": true },
    { "logicalName": "sprk_status", "type": "Picklist", "options": [{"value":1,"label":"Open"},{"value":2,"label":"Closed"}] },
    { "logicalName": "sprk_clientid", "type": "Lookup", "targets": ["account"] },
    ...
  ]
}
```

**Map types to factory defaults**:

| Dataverse type | Factory default | Notes |
|---|---|---|
| `String` (text) | `"Sample <field>"` | Use realistic placeholder, not Lorem |
| `Memo` (multi-line) | `"Auto-generated by prototype seed factory."` | |
| `DateTime` | `new Date(Date.now() + N * 86_400_000).toISOString()` | Spread across realistic windows |
| `Integer` / `BigInt` | `N * 10` (predictable progression) | |
| `Decimal` / `Money` | `1000 + i * 250` | |
| `Boolean` (twoOptions) | `false` (or alternate) | |
| `Picklist` (optionset) | First active option's value | |
| `State` / `Status` | `0 / 1` (active by default) | |
| `Lookup` | `null` (or a GUID from a related factory if also seeded) | |
| `Uniqueidentifier` (PK) | `generateGuid()` | Always overridable |
| `EntityCollection` (M:N) | `[]` | Empty unless explicitly seeded |
| `OwnerType` | `PROTOTYPE_USER_ID` (`"00000000-0000-0000-0000-000000000001"`) | |

### Step 3: Generate Factory File

Write to `c:/code_files/spaarke-prototype/_infra/seed/factories/sprk_<entity>.ts`:

```typescript
/**
 * sprk_<entity> factory.
 *
 * Produces records matching the Dataverse schema for `sprk_<entity>` that
 * consumer harnesses query. Fields are populated with realistic defaults;
 * any override can replace them.
 *
 * # Schema reference (what production code reads/writes)
 *
 * - `sprk_<entity>id` (GUID, primary key)
 * - `sprk_name` (string, primary name)
 * - `<other fields>` ...
 *
 * # Sandbox-only
 *
 * This file lives in spaarke-prototype/_infra/seed/. Production code never
 * imports from here. The data is mock-only.
 */

/* eslint-disable @typescript-eslint/no-explicit-any */

import type { SeedStore } from "../../mocks/xrm";

const PROTOTYPE_USER_ID = "00000000-0000-0000-0000-000000000001";

export interface Sprk<EntityName>Overrides {
  sprk_<entity>id?: string;
  sprk_name?: string;
  // ... one field per attribute, all optional
  createdon?: string;
  modifiedon?: string;
  ownerid?: string;
}

function generateGuid(): string {
  return "xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx".replace(/[xy]/g, (c) => {
    const r = (Math.random() * 16) | 0;
    const v = c === "x" ? r : (r & 0x3) | 0x8;
    return v.toString(16);
  });
}

export function makeSprk<EntityName>(overrides: Sprk<EntityName>Overrides = {}): any {
  const now = new Date();
  return {
    sprk_<entity>id: overrides.sprk_<entity>id ?? generateGuid(),
    sprk_name: overrides.sprk_name ?? "Sample <Entity>",
    // ... one field per attribute, with sensible default
    statecode: 0,
    statuscode: 1,
    ownerid: overrides.ownerid ?? PROTOTYPE_USER_ID,
    createdon: overrides.createdon ?? now.toISOString(),
    modifiedon: overrides.modifiedon ?? now.toISOString(),
    ...overrides,
  };
}

export function seedSprk<EntityName>s(count = 10): any[] {
  return Array.from({ length: count }, (_, i) =>
    makeSprk<EntityName>({
      sprk_name: `${SAMPLE_NAMES[i % SAMPLE_NAMES.length]}`,
      // ... per-index variation for visual realism
    }),
  );
}

export function installSprk<EntityName>s(store: SeedStore, count = 10): void {
  if (!store.records.sprk_<entity>) {
    store.records.sprk_<entity> = [];
  }
  store.records.sprk_<entity>.push(...seedSprk<EntityName>s(count));
}

// ── Realistic content (helps the prototype feel like real data) ──────────────

const SAMPLE_NAMES = [
  // 5-10 realistic legal-domain examples for this entity type
  "...",
];
```

**Replace `<EntityName>` with the PascalCase form** (e.g., `Matter`, `Event`, `Communication`).

**Realistic content**: spend extra effort on the sample names — they're what makes the harness feel like a real environment. For legal entities, use bar-passed-attorney-quality examples:
- sprk_matter: "Smith v. Jones — Wrongful Termination", "Acme Co. — Corporate Reorganization"
- sprk_event: "Discovery deadline — Acme matter", "Settlement conference — Smith v. Jones"
- sprk_invoice: "Q2 2026 billing — Acme Corp.", "Pro bono — Jefferson Society"

### Step 4: Register in `_infra/seed/index.ts`

Add an `export` line:

```typescript
export {
  makeSprk<EntityName>,
  seedSprk<EntityName>s,
  installSprk<EntityName>s,
  type Sprk<EntityName>Overrides,
} from "./factories/sprk_<entity>";
```

### Step 5: Extend a Preset (Optional)

**IF the user named a preset to extend**:

Open `c:/code_files/spaarke-prototype/_infra/seed/presets/<preset>.ts` and add:

```typescript
import { seedSprk<EntityName>s } from "../factories/sprk_<entity>";

export function <presetName>() {
  return {
    records: {
      // ... existing entities
      sprk_<entity>: seedSprk<EntityName>s(<count>),
    },
  };
}
```

**IF the user said "new" preset**:

Create `_infra/seed/presets/<new-preset>.ts` with the same shape; then add a `case '<new-preset>': return <presetName>();` to the `createSeed` switch in `_infra/seed/index.ts`.

## Anti-Patterns (do NOT do)

| ❌ Don't | Why |
|---|---|
| Copy production schema files (entity metadata XML) into the prototype | Adds maintenance burden; rely on Dataverse MCP for live schema |
| Generate factories with Lorem-ipsum placeholders | Realistic content is what makes the harness valuable for UX work |
| Hard-code GUIDs that look like real Dataverse IDs | Use `generateGuid()`; if you need a stable ID for cross-referencing, document it |
| Add the new factory's data to ALL existing presets automatically | Each preset has a deliberate scope; let the user choose which to extend |
| Skip the realistic SAMPLE_NAMES section | A factory with placeholder names defeats the purpose — the prototype should feel like a real environment |
| Import from production source (`@spaarke/*`) inside a seed factory | Factories must be self-contained; cross-coupling makes them fragile |

## Failure Modes & Recovery

| Symptom | Cause | Fix |
|---|---|---|
| Dataverse MCP returns "table not found" | Entity not yet published to the connected environment | Fall back to user-supplied field list; note in the factory's docstring that schema may drift |
| Factory generated but harness still shows empty list | Mock OData filter rejects the query | Check `[xrm-mock]` console warnings; the mock's filter parser supports `eq/ne/lt/gt/le/ge/and/or` but not `contains/startswith` — adjust query OR extend `_infra/mocks/xrm.ts` |
| TypeScript error: `Property 'sprk_X' does not exist on type 'SeedStore'` | `SeedStore` typing is open via index signature — this usually means a typo in the factory's `installSprk*` function | Verify the entity name matches across `store.records.<name>` and the imports |
| Preset registered but consumer harness can't find it | Missing case in the `createSeed` switch | Add `case '<preset>': return <presetFn>()` in `_infra/seed/index.ts` |
| Lookup field shows as null in UI | Factory sets lookup to `null`; the consumer's UI doesn't have a "no parent" branch | Either seed a related entity AND set the lookup to its GUID, OR fix the UI to handle null gracefully (which is closer to real-world behaviour anyway) |
