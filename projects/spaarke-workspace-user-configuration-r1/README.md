## Problem Statement

Today the workspace layout is hardcoded — every user sees the same 3-row, 5-section dashboard. No way to add/remove/reorder sections, no named workspaces, no personalization. The `WorkspaceShell` component already accepts declarative `WorkspaceConfig`, but no mechanism exists for users to create or modify that configuration at runtime.

## Solution

Allow users to personalize their workspace by:
1. Selecting a layout template (6 predefined grid options)
2. Choosing which sections to display (from a code-side Section Registry)
3. Arranging sections via drag-and-drop in a 3-step wizard
4. Creating multiple named workspaces with a default
5. Storing layout configuration per-user in Dataverse (`sprk_workspacelayout`)

## Scope

### Deliverables
- Dataverse entity `sprk_workspacelayout`
- 8 BFF CRUD endpoints for workspace layout management
- Section Registry with 5 initial entries + standard `SectionFactoryContext`
- Layout Wizard Code Page (`sprk_workspacelayoutwizard`) — React 19, Vite single-file
- Workspace Header with dropdown switcher + settings
- Dynamic config builder (layout JSON + registry factories -> WorkspaceConfig)
- 6 predefined layout templates
- `useFeedTodoSync` no-op fix for SmartToDo independence
- Loading states, sessionStorage caching, URL deep-linking

### Out of Scope
- Admin-level layout management, section-level data filtering, mobile layouts
- Layout sharing, role-based defaults, user-adjustable section heights

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Section Registry pattern | Single source of truth for available sections; new sections = 1 registration |
| Store IDs + positions only | No JSX serialized to Dataverse; registry handles rendering |
| System vs User workspaces | System workspaces always available as safety net; Save As for customization |
| Code Page for wizard | React 19 + Vite single-file build per ADR-026; launched via navigateTo |
| useFeedTodoSync no-op | SmartToDo must work independently in any context |
