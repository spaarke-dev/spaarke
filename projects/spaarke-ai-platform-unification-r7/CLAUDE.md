## Key Technical Constraints

### From spec.md / design.md

- **Single-hop dispatch**: `PlaybookOrchestrationService.ExecuteNodeAsync` MUST read `node.sprk_executortype` directly. No lookup chain. No structural fallback. No Action ActionType override branch.
- **No transition mode / backward-compat shim**: big-bang cutover per Q6 RESOLVED.
- **AiCompletionNodeExecutor pattern**: model on `EntityNameValidatorNodeExecutor` (Singleton, ILogger only, ConfigJson read + Validate). Reuses `PromptSchemaOverrideMerger` for per-node prompt overrides.
- **Action FK becomes optional** at schema level; required at executor `Validate()` for prompt-driven executors (AiAnalysis, AiCompletion, AiEmbedding); prohibited for pure executors (Condition, Start, ReturnResponse, etc.)
- **Documentation discipline**: DELETE outdated content, UPDATE current. NO SUPERSEDED markers, NO redirect stubs, NO archive folders. PR review gates drift.
- **Scope boundary**: `docs/architecture/ai-architecture-consumer-routing.md` is READ-ONLY (chat-routing-redesign-r1 owns). `docs/data-model/sprk_playbookconsumer.md` is NOT created by R7.

### From CLAUDE.md / ADRs

- **BFF Hygiene §10**: every BFF-touching task runs `.claude/constraints/bff-extensions.md` checklist + states placement decision + verifies publish-size impact (NFR-01 ceiling ≤+2 MB, BFF total ceiling 60 MB compressed)
- **Component Justification §11**: every new component answers Existing / Extension / Cost-of-doing-nothing. Tasks that ONLY modify existing files are exempt.
- **ADR-010 DI Minimalism**: AiCompletionNodeExecutor registered as Singleton via existing `AnalysisServicesModule.AddNodeExecutors` pattern
- **ADR-013 BFF AI Architecture**: `IInvokePlaybookAi` triangle stays canonical; R7 deepens, doesn't replace
- **ADR-029 BFF Publish Hygiene**: per-task publish-size + CVE verification on every R7-touching task
- **ADR-038 Testing Strategy**: integration-heavy pyramid; ban `Mock<HttpMessageHandler>` + DI-registration tests + ctor null-check tests. AiCompletionNodeExecutor unit tests mock `IOpenAiClient` only.

### Rigor levels per task (auto-detected by task-execute)

- **FULL rigor** (code-review + adr-check at Step 9.5): tasks tagged `bff-api`, `code-impl`, `dataverse-schema`, `pcf`, or `code-page`; modifying `.cs`/`.ts`/`.tsx`/`.ps1`; 6+ steps
- **STANDARD rigor**: doc tasks, audit tasks, planning tasks
- **MINIMAL rigor**: simple updates, inventory tasks
- **TEST-MODIFYING override** (binding per ADR-038): any task modifying `tests/**` runs FULL rigor unconditionally
