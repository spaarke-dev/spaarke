## Residual Cluster B disposition (37 tests still failing)

These tests were previously masked by host-build failures. Now that the host starts cleanly, the underlying production / wiring issues surface. Distribution:

| Test class | Failures | Top symptoms | Classification |
|---|---:|---|---|
| `KnowledgeBaseEndpointsTests` | 13 | `Failure to infer one or more parameters` (minimal-API endpoint metadata gen) | Wave 2.5 / new follow-up — likely needs additional DI stubs |
| `ChatEndpointsTests` | 11 | `Failure to infer one or more parameters` + Moq verification | Wave 2.5 / new follow-up — same root cause as KB |
| `ReAnalysisFlowTests` | 8 | `Failure to infer one or more parameters` + SSE stream errors | Wave 2.5 / new follow-up |
| `AuthorizationTestFixture` | 5 | `Failure to infer one or more parameters` (Analysis:Enabled=false breaks param binding) | Wave 2.5 / new follow-up |

**Root cause hypothesis** (same pattern across all 4 fixtures): each sets `Analysis:Enabled=false` and `DocumentIntelligence:Enabled=false` to skip AI module registration, but the corresponding endpoints (KB, Chat, ReAnalysis, Authorization) are unconditionally mapped in `Program.cs`. Endpoint parameter inference fails at startup because required services aren't in DI. The fixtures already stub many such services — but apparently NEW services were added to `Program.cs` after the fixtures were written (drift signal). **This is exactly the anti-drift use case the `bff-extensions.md` "Test update obligation" addresses (Phase 4 task 080).**

**Per POML Step 8 / §6.2**: residuals are NOT this task's scope (this task absorbed the config-cluster). Triage tagging deferred to a Wave 2.5 follow-up task or to integration LOW-tier P23.L. Do NOT mark these tests `repaired` / `real-bug-pending-fix` / `flaky-quarantined` in this task — they remain in `in-progress-task-T-XX` intermediate state until follow-up triages them.
