## Overview

Spaarke Compose is the AI-native legal drafting workspace. Compose and Microsoft Word are complementary surfaces (handoff model, not competition): Compose owns AI-driven legal work; Word remains the advanced word processor reached via SharePoint Embedded. **R1 delivers the foundation** — workspace layout + three-pane wiring + TipTap (out-of-box) editor + SPE plumbing + ChatSession reuse + a consumer-routing smoke-test playbook dispatch.

## Problem Statement

Spaarke today lacks an AI-native drafting surface inside the legal workspace. Existing options force users to choose between:

1. **Microsoft Word** — strong word-processing tooling, but no AI-driven legal work (clause explanation, playbook comparison, draft generation, derived-insight retention).
2. **SpaarkeAi ConversationPane** — strong AI conversation surface, but no first-class document editor; legal drafting flow is bolted onto chat rather than a true editor.

The result: legal-drafting users either leave the Spaarke surface entirely (losing matter context, derived insights, and AI continuity) or work around the absence with copy/paste between chat and Word. There is no single surface that delivers *both* AI orchestration *and* a real document editor coordinated with matter context.

## Solution Summary

Spaarke Compose ships as a **new workspace layout** in SpaarkeAi (`sprk_workspacelayout` system record, template `single-column`, section `compose-editor`). The Workspace pane hosts a **TipTap-based React editor** (out-of-box feature set only — advanced features like tracked changes and footnotes live in Word via handoff). The Assistant pane (existing `ConversationPane`) and Context pane (existing matter intelligence surface) wrap Compose with six coordinated data flows. Documents live in **SharePoint Embedded** (always file-of-record); ephemeral uploads become `sprk_document` records on first Save. AI dispatch reuses the existing `ChatSession` three-tier persistence + `IConsumerRoutingService`/`IInvokePlaybookAi` facades — Compose adds two new JPS scopes (`compose-selection`, `compose-document`) and one consumer type (`compose-summarize`) wired to the existing **Document Summary** playbook as an end-to-end smoke test. **Word handoff** reuses the existing `GET /api/documents/{id}/open-links` endpoint; the client `useDocumentActions` hook is extracted to a new `@spaarke/document-operations` shared library so both SemanticSearch and Compose consume from one source.

## Scope

### In Scope (R1)

- New `sprk_workspacelayout` system record for Compose (label: "Compose")
- Three-pane coordination wiring (six flows, locked TypeScript data contracts; receivers may be stubs)
- TipTap React editor shell mounted in Workspace pane via new `compose-editor` section type
- TipTap OOB feature set only (no custom advanced-feature integration)
- SPE plumbing — load DOCX from SPE, save edits back as new SPE versions; SPE always file-of-record
- Upload-from-Assistant path (Path B) — Assistant file upload → SPE → Compose mount (ephemeral)
- Document promotion on first Save (creates `sprk_document` record; idempotent)
- ChatSession three-tier reuse (Redis/Cosmos/Dataverse) with `DocumentId` binding — NO new entity
- Two new JPS scopes (`compose-selection`, `compose-document`)
- Consumer-routing smoke test: `compose-summarize` → Document Summary playbook (existing id `47686eb1-9916-f111-8343-7c1e520aa4df`) wired end-to-end
- Compose → Word handoff (Web + Desktop) reusing existing `/api/documents/{id}/open-links` + `DesktopUrlBuilder`
- Shared library `@spaarke/document-operations` extraction (from SemanticSearch's `useDocumentActions`)
- Per-user single-session SPE check-out lock + multi-tab conflict UX + 15-min orphan-lock release
- Empty-state UX (Browse / Search options)
- Path A entry: command-bar "Open in Compose" → modal shell with full-screen toggle
- 7 new BFF endpoints under `/api/compose/` (upload, load, save, promote, checkout, checkin, dispatch-action)
- Unit tests for each new BFF service per CLAUDE.md §10 #6

### Out of Scope (R1 — deferred)

- AI actions on selection (Explain clause / Replace / Compare-to-playbook / Draft alternative) — **R2**
- Rich session-memory content beyond `DocumentId` binding (anchored annotations, action log, derived insights) — **R2**
- Return-from-Word round-trip annotation re-anchoring + conflict UX banner — **R2**
- DOCX subset *enforcement* (R1 publishes a draft subset spec; enforcement code = R2)
- Office Add-in entry path (Word → Compose) — **R3**
- PDF / email / transcript artifact types — **R4+**
- Real-time multi-user co-editing (CRDT) — **R5+**
- Tracked changes round-trip with Word — **never (out of architecture)**
- Comments stored as Word `<w:comment>` elements — **never** (Compose stores comments as ChatSession annotations in R2+)

## Key Decisions

| Decision | Rationale | Reference |
|----------|-----------|-----------|
| Editor: TipTap (ProseMirror) | OOB feature richness; React-native; well-maintained; supports DOCX bridges | design.md §14 #1 |
| OOB-only scope | Avoids custom-integration cost for tracked changes / footnotes / fields / equations; handoff to Word for those | spec.md FR-03 + §Owner Clarifications |
| ChatSession reuse (no new entity) | Modern three-tier pattern (Redis/Cosmos/Dataverse) already proven; `sprk_analysis.sprk_chathistory` is superseded | spec.md §Supersession Map + design.md §6 |
| BFF placement (all in `Sprk.Bff.Api`) | All R1 endpoints touch SPE/Graph + Dataverse; no plugin / microservice justified | spec.md §Placement Justification |
| AI access via PublicContracts facade | Refined ADR-013 (2026-05-20) — no direct injection of AI internals into Compose CRUD code | [ADR-013](../../docs/adr/) |
| First consumer type: `compose-summarize` → Document Summary playbook | Simplest E2E smoke test; reuses existing summarize playbook patterns; predictable success criteria | design.md §14 #6 |
| Per-user single-session lock via SPE check-out | Avoids "user collaborating with themselves" failure mode | design.md §14 #4 |
| Path A entry: modal with full-screen toggle | Reuses existing SpaarkeAi modal pattern (ConversationPane, ContextPaneController, launch-resolver) | design.md §14 #3 |
| Empty state: Browse + Search | Two-action default-open per design.md §14 #5 | design.md §14 #5 |
| First-Save promotion (idempotent) | Ephemeral docs become `sprk_document` records on user-explicit Save | design.md §14 + FR-06 |

| Risk | Impact | Likelihood | Mitigation |

| SPE check-out lock + heartbeat reliability (orphan locks) | High | Low | **Spike #3** validates check-out/check-in flow + heartbeat (default 15 min idle); refinable post-spike |

| Publish-size delta exceeds +2 MB / approaches 60 MB ceiling | Low | Low | Measure `dotnet publish -c Release` per BFF-touching task; TipTap + DOCX bridge are client-side (no server delta) |
