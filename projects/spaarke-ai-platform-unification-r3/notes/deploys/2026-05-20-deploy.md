## Supplemental Deploy ‚Äî Task 088 (embed wiring fixes 4.1) ‚Äî 2026-05-21

### Why

Task 087 made the SpaarkeAi `WorkspacePaneMenu` "Switch Workspace" dropdown actually open the chosen workspace as a tab via embedded `LegalWorkspaceApp`. Operator smoke immediately after deploy surfaced three wiring bugs that the embed had exposed:

**Bug 1 ‚Äî all tabs show the same workspace**: whether the operator picked "Ralph Workspace 4", "Ralph Workspace 5", or "Corporate Workspace" from the dropdown, every tab rendered the user's last-active workspace (e.g. "Corporate Workspace"). The `initialWorkspaceId` prop was being passed correctly through `LegalWorkspaceApp ‚Üí WorkspaceGrid ‚Üí useWorkspaceLayouts(initialWorkspaceId)`, but the hook's cache-first hydration path read from `sessionStorage` BEFORE applying `initialWorkspaceId`. Because the `sessionStorage` cache keys (`sprk:workspace:activeLayout` / `sprk:workspace:layoutsList`) are GLOBAL across all embedded tabs, every new tab hydrated from the same cached "last-active" value. The API path also wrote back to the same cache after resolution, so each newly opened tab clobbered the previous one's view.

**Bug 2 ‚Äî every tab title is "Workspace"**: the embedded `WorkspaceLayoutWidget` widget type is registered with `displayName: "Workspace"` in `WorkspaceWidgetRegistry`. `WorkspacePane.tsx`'s `widget_load` handler uses a precedence ladder (event.displayName ‚Üí registry metadata.displayName ‚Üí widgetType) to pick the tab title. `WorkspacePaneMenu.handleLayoutSelect` was passing the friendly layout name inside `widgetData.layoutName` (a payload field) but NOT setting `displayName` at the top level of the dispatched event, so the ladder fell through to the registry metadata's generic label "Workspace".

**Bug 3 ‚Äî document preview throws "Runtime config not initialized"**: clicking the document preview action inside an embedded LegalWorkspace tab threw `Uncaught Error: [LegalWorkspace] Runtime config not initialized. Call initRuntimeConfig() in main.tsx before using getBffBaseUrl().` LegalWorkspace has its OWN `runtimeConfig.ts` singleton in `src/solutions/LegalWorkspace/src/config/` ‚Äî separate from SpaarkeAi's `src/solutions/SpaarkeAi/src/config/runtimeConfig.ts` singleton. SpaarkeAi's `main.tsx` bootstrap called SpaarkeAi's own `setRuntimeConfig(config)` but NOT LegalWorkspace's. Any code path inside the embedded tree that called `getBffBaseUrl()` (document preview's `Xrm.Navigation.navigateTo` handler, the BFF fetch surface) hit an uninitialized singleton.

### What changed

Three surgical fixes, each isolated to embedded-mode code paths so standalone LegalWorkspace remains byte-identical (FR-25 / NFR-10):

**Fix 1 (Bug 1)** ‚Äî `src/solutions/LegalWorkspace/src/hooks/useWorkspaceLayouts.ts`:

- Added new `UseWorkspaceLayoutsOptions` interface with `initialWorkspaceId?: string` + `embedded?: boolean`.

## Task 100 ‚Äî Collapsed strip top-alignment + empty-pane width redistribution

> **Task**: 100 ‚Äî Collapsed strip top-alignment + empty-pane width redistribution
> **Phase**: F (Verification ‚Äî remediation, Round 7 operator smoke 2026-05-22)
> **Verdict**: **PASS ‚Äî operator feedback two-item resolution (icon vertical-alignment + freed-width fill)**
> **Operator**: Claude Code (task-execute, FULL rigor)
> **Branch**: `work/spaarke-ai-platform-unification-r3`

### Operator feedback driving this task (Round 7, 2026-05-22)

Two items from operator smoke after the Wave-2/3 deploy (096/097/098/099/093):

1. "The collapsed pane icons should be **top aligned**, not middle aligned."
2. "If Assistant pane AND Workspace pane are collapsed, the **container for Context pane should extend to full width** of the main page container with the background color." Implicit extension: same applies to ANY two-collapsed combination (the remaining expanded pane should fill the freed width).

### Investigation

`src/client/shared/Spaarke.UI.Components/src/components/ThreePaneLayout/ThreePaneLayout.tsx` end-to-end review. Two root causes identified:

**(1) Icon vertical centering** ‚Äî the `collapsedIcon` style (task 096, lines 183-195) had `flexGrow: 1` set; the icon container therefore stretched to fill the entire remaining vertical space of its column-flex parent, and the `alignItems: 'center' + justifyContent: 'center'` inside that stretched container centered the icon vertically. Visual result: icon floats roughly in the middle of the strip.

**(2) Empty pane width** ‚Äî the layout algorithm used `style={{ width: ${leftWidthPx}px }}` for the left pane and `style={{ width: ${rightWidthPx}px }}` for the right pane (fixed pixel widths). The center pane has `flex: 1` in `styles.centerPane`. When ALL three panes are expanded ‚Üí center fills (correct). When center is collapsed (e.g. operator's case: Assistant + Workspace collapsed ‚Üí only Context expanded), the right pane stayed at its fixed `rightWidthPx` (400px default) and the remaining ~containerWidth ‚àí 96 ‚àí 400 was an empty `colorNeutralBackground1` strip at the right edge.

### Implementation

#### A. Icon top-alignment

(1) Removed `flexGrow: 1` from `styles.collapsedIcon` so the icon no longer stretches to fill the strip ‚Äî it sits naturally below the parent's `paddingTop`.

(2) Added `justifyContent: 'flex-start'` and bumped `paddingTop: tokens.spacingVerticalS` ‚Üí `tokens.spacingVerticalM` on all three `*PaneCollapsed` styles (`leftPaneCollapsed`, `rightPaneCollapsed`, `centerPaneCollapsed`). `spacingVerticalM` (~12px) matches the visual offset of the expanded PaneHeader's icon: PaneHeader uses `paddingTop: spacingVerticalS` (~8px) inside a `minHeight: 40px` row, which visually places its icon ~12px below the top edge. Matching that rhythm means the icon doesn't visually shift up/down when the pane toggles between expanded and collapsed.

All three changes are token-based ‚Äî no hex / rgba / px literals (ADR-021).

#### B. Width-redistribution algorithm

Switched the left/right pane render from fixed `width` to a conditional `flex: 1 1 auto` based on a new `expandedCount` computation. Algorithm:

| Visibility state | Left pane | Center pane | Right pane | Operator-visible result |
|---|---|---|---|---|
| All 3 expanded | fixed `leftWidthPx` | `flex: 1` (centerPane CSS) | fixed `rightWidthPx` | Center fills (unchanged baseline) |
| Left collapsed only | `0 0 48px` | flex 1 | fixed | Center fills (already worked) |
| Right collapsed only | fixed | flex 1 | `0 0 48px` | Center fills (already worked) |
| Center collapsed only | fixed | `0 0 48px` | **flex 1 1 auto (NEW)** | Right (Context) fills the freed center+right space |
| Left + Center collapsed | `0 0 48px` | `0 0 48px` | **flex 1 1 auto (NEW)** | Right (Context) fills (operator's primary case) |
| Left + Right collapsed | `0 0 48px` | `flex 1` (centerPane CSS) | `0 0 48px` | Center (Workspace) fills |
| Center + Right collapsed | **flex 1 1 auto (NEW)** | `0 0 48px` | `0 0 48px` | Left (Assistant) fills |
| All 3 collapsed | `0 0 48px` | `0 0 48px` | `0 0 48px` | Three 48px strips (degenerate; acceptable no-op) |

Two new derived booleans drive the conditional `style`:

```ts
const expandedCount = (isLeftVisible ? 1 : 0) + (isCenterVisible ? 1 : 0) + (isRightVisible ? 1 : 0);
const leftIsSoleExpanded = isLeftVisible && expandedCount === 1;
const rightIsSoleExpanded = isRightVisible && expandedCount === 1;
const rightFillsForCollapsedCenter = isLeftVisible && !isCenterVisible && isRightVisible;
```

Left pane render: `leftIsSoleExpanded ? { flex: '1 1 auto', width: 'auto' } : { width: ${leftWidthPx}px }`.
Right pane render: `(rightIsSoleExpanded || rightFillsForCollapsedCenter) ? { flex: '1 1 auto', width: 'auto' } : { width: ${rightWidthPx}px }`.

The center pane was already `flex: 1` via `styles.centerPane` ‚Äî no change needed; it naturally fills whenever it's expanded.

**Backgrounds**: each expanded pane component (`ConversationPane`, `WorkspacePane`, `ContextPaneController`) already renders at `height: 100%; width: 100%` with its own `colorNeutralBackground1` body, so the expanded pane's own background fills the freed width automatically ‚Äî no shared-lib background change needed.

**Splitters**: PanelSplitters between adjacent panes are hidden when either neighbor is collapsed (task 094 logic ‚Äî `isLeftVisible && isCenterVisible` / `isRightVisible && isCenterVisible`). No change needed.

#### What did NOT change

- SpaarkeAi pane components (ConversationPane / WorkspacePane / ContextPaneController / ThreePaneShell) ‚Äî pure shared-lib CSS-driven change.
- `ThreePaneLayout.types.ts` ‚Äî no new props.
- `useThreePaneLayout` hook ‚Äî width persistence logic untouched; the new flex behavior overrides the persisted `leftWidthPx`/`rightWidthPx` only in the sole-expanded / center-collapsed cases.
- LegalWorkspace standalone ‚Äî does not import `ThreePaneLayout`; bundle byte-identical (FR-25 / NFR-10 preserved).

Five surgical edits: three `*PaneCollapsed` styles (added `justifyContent: 'flex-start'` + bumped `paddingTop` token); one `collapsedIcon` style (removed `flexGrow: 1`); one render-block transformation (new `expandedCount` + `leftIsSoleExpanded` / `rightIsSoleExpanded` / `rightFillsForCollapsedCenter` derivations + conditional `style` on left/right pane wrappers).
