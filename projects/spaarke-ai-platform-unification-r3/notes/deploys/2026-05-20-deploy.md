- The effect's dep array was `[fetchKey]` only — it had **no readiness signal** to re-fire on once auth/config bootstrapped. The menu was therefore permanently empty for the lifetime of the session.

The established R3 pattern (used by `WorkspaceHomeTab`, `HistoryOverlay`, `useDailyBriefing`) is to source `authenticatedFetch` + `bffBaseUrl` + `isAuthenticated` from `useAiSession()`, which auto-defers consumers until auth is ready.

| Change | Detail |

| Import surface | Removed module-level `authenticatedFetch` (from `@spaarke/auth`) + `getBffBaseUrl` (from `../../config/runtimeConfig`). Added `useAiSession` (from `@spaarke/ai-widgets`). Kept `buildBffApiUrl` from `@spaarke/auth` (it's pure and doesn't depend on auth state). |

**Trigger**: Operator confirmation that Daily Briefing loads correctly in the standalone Daily Briefing Code Page but renders the "Nothing to see right now — enjoy your day" empty state in the SpaarkeAi Workspace Home tab on cold load. Operator direction: "having the visuals on the screen but not actually [working] is not the finished product" + "all of this needs to use all of our existing auth, bff.api connectors, etc." Root cause confirmed by App Insights: the SpaarkeAi WorkspaceHomeTab auto-fired the shared `useDailyBriefing` hook with `buildEmptyNarrateRequest()` (task 069), so the BFF returned 200/empty bullets (task 083) and the embedded section rendered the empty state on every cold load — even when the user had real notifications.

**Task**: 086 — corrective fix (Round 4 Fix 3 of 3).

### Investigation: where does the standalone get its data?

Located the standalone Daily Briefing Code Page at `src/solutions/DailyBriefing/`. Key files:

- `src/solutions/DailyBriefing/src/services/notificationService.ts` — queries `appnotification` entity via `Xrm.WebApi.retrieveMultipleRecords`, parses the `data` JSON column for `customData.category` / `priority` / `regardingName` etc., groups by category into `ChannelFetchResult[]`.
- `src/solutions/DailyBriefing/src/services/briefingService.ts#buildNarrationRequest` — converts `ChannelFetchResult[]` into the `NarrateRequest` envelope (categories + priorityItems + per-channel items) and POSTs to `/api/ai/daily-briefing/narrate`.
- `src/solutions/DailyBriefing/src/hooks/useNotificationData.ts` — orchestrates the Xrm fetch on mount.
- `src/solutions/DailyBriefing/src/App.tsx` — composes `useNotificationData` + `useBriefingNarration` into the digest UI. NOTE: the App auto-loads on mount; no user input needed.

The standalone is fully programmatic — it doesn't need user input to generate a briefing. It just (a) resolves Xrm via frame-walking, (b) queries `appnotification`, (c) groups + builds the payload, (d) POSTs to narrate.

### Strategy: Option H (Hoist) augmentation + Option C (Copy) loader

Per task 084 precedent (Round 4 Fix 1):

1. Hoist an optional `loadNotificationContext` callback parameter into the shared `useDailyBriefing` hook + `createDailyBriefingRegistration` factory + `DailyBriefingSection` props. When supplied, the hook calls it before POST-ing to narrate and forwards the populated payload verbatim. When omitted, the legacy empty-payload contract is preserved (BFF returns empty bullets — task 083 graceful path).
2. Copy the standalone Daily Briefing's notification-context logic into a new SpaarkeAi-local service `src/solutions/SpaarkeAi/src/services/notificationContextLoader.ts`. The copy uses the SAME Xrm.WebApi query, the SAME `customData` parsing, the SAME `groupByCategory` logic, and the SAME `buildNarrationRequest` payload assembly as the standalone. Carries a JSDoc cross-reference to the source files for sync.
3. Wire the loader into `WorkspaceHomeTab.tsx` by passing it to the `createDailyBriefingRegistration` factory.

Why Copy (not Hoist) for the loader logic itself: the standalone code page's `useNotificationData` hook tangles together notifications + preferences + mark-as-read + refresh actions that SpaarkeAi's Home tab doesn't need. A clean hoist would require splitting that hook apart — out of scope for a corrective fix. The copy is ~280 lines and lives entirely in SpaarkeAi (ADR-012: Dataverse entity strings like `appnotification`, `customData`, `toasttype` belong solution-local, not in `@spaarke/ui-components`).

LegalWorkspace's existing `dailyBriefing.registration.ts` shim does NOT supply the new `loadNotificationContext` prop, so its standalone behavior is byte-equivalent at the runtime level (the only delta is the optional parameter forwarding through the hook — ~140 bytes of code path that's `undefined` at runtime for LegalWorkspace). FR-25 / NFR-10 preserved at the behavior level. The standalone Daily Briefing Code Page is untouched.
