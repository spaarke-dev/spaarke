# Lessons learned — ai-spaarke-ai-workspace-UI-r2

**Project close**: 2026-07-01
**Duration**: Single continuous autonomous execution session (~4 hours real time; ~19 hours estimated in plan.md)
**Scope delivered**: 21 FRs + 9 NFRs across 3 phased milestones (framework + Communications + SmartTodoModal retirement + 5 doc surfaces)

## What worked

- **Framework-level change beats per-widget change.** Task 002 modified `DataGrid.defaultRecordOpen` and — because none of the 5 workspace widgets pass an `onRecordOpen` override — the change propagated to Documents, Matters, Projects, Invoices, and Work Assignments with zero per-widget code. FR-04 through FR-08 were satisfied by ONE code change plus static verification. This is the kind of surface consolidation that FR-consolidation payoff makes visible.
- **Pure-helper extraction as test seam.** Both Phase 1 (`buildRecordOpenNavArgs` in `DataGrid.tsx`) and Phase 3 (`openSprkTodoAsLayout1` in `SmartTodoApp.tsx`) followed the same pattern: extract a module-scope pure function that constructs the `Xrm.Navigation.navigateTo` payload, keep it exported for testing but NOT re-exported from any barrel (matches the existing `ColumnFilterType`/`ColumnFilterOption` pattern). Nine focused unit tests in Phase 1 covered every FR acceptance criterion.
- **Pattern D dual-use replication for Communications.** Task 011 + 012 executed cleanly because the 5 existing entity-view widgets had already established the shim shape. Communications took ~1 hour of code from template inheritance; new-file authoring didn't require any novel design decisions.
- **Retirement audit before migration.** Task 020's callsite enumeration (`notes/smart-todo-modal-callsites.md`) revealed that all `SmartTodoModal` open triggers converged on ONE `modalTodoId` React state variable via ONE listener + ONE useEffect. Migration Pattern A (change the listener, not the dispatchers) was ~30 lines of diff instead of touching every dispatcher.
- **Grep-verifiable success criteria pay off.** 10 of 14 success criteria were static-verifiable (grep for `window.open`, `value: 85`, `main.aspx`, `SmartTodoModal`, etc.). The 4 that require manual QA are documented with clear post-deploy checklists in [`phase1-verification.md`](phase1-verification.md) and [`success-criteria-verification.md`](success-criteria-verification.md).

## What surprised

- **The R4 iframe hybrid was more entangled than expected in comments.** The literal code deletion in task 022 was 4 files. But grep for `SmartTodoModal` found 10+ files with docstring references (`SmartTodoApp.tsx` line 405 alone had a stale R4 comment). Docstring cleanup during migration is worth budgeting explicitly — not free.
- **`useLaunchContext.ts` is NOT iframe-specific.** Initial assumption from task 020 was that `useLaunchContext.ts` might be deletable. On inspection, it's a broader URL-launch-context parser handling 3 discriminators (`createTodo`, `openTodos`, `openTodo`); only `openTodo` was iframe-adjacent. Kept the hook; migrated the `openTodo`-handling `useEffect` in `SmartTodoApp.tsx` to call `openSprkTodoAsLayout1` instead of `setModalTodoId`. The `openTodo` discriminator is now dead-in-practice (no emitter in `src/`), but the parser contract remains for URL-launch portability.
- **The R4 modal's `< / >` record-navigation feature dropped intentionally.** The R4 `<SmartTodoModal>` supported walking the current filter set. Layout 1 (OOB modal) doesn't. Spec FR-13 explicitly accepted this trade-off; users navigating between to-dos will return to the workspace and pick the next one. Worth flagging as a UX regression to inform post-deploy user feedback loops.
- **LegalWorkspace's `handleOpenTodo` was already at 80% × 80% via UAT 2026-06-21 round 5.** R2 task 021 discovered that the LegalWorkspace path already used `Xrm.Navigation.navigateTo` (not the iframe modal) — but at 80%, not FR-20's binding 85%. So the "migration" for that path was a 1-character diff (80 → 85). The retirement decision + FR-20 binding aligned these two divergent behaviors that had drifted independently. Doc-standardization pays for itself in cases like this.

## What would you do differently

- **Split PR #530 by phase.** The project spec + CLAUDE.md said "do NOT batch phases into a single PR." Combining all three phases (Phase 1 framework + Phase 2 Communications + Phase 3 SmartTodoModal retirement) into PR #530 was an operator mid-session choice ("don't wait for CI — proceed"). It saved wall-clock time but hurts review quality — reviewers now have to context-switch across three separate concerns in one diff. If future projects do the same, plan for the reviewer to split at review time, OR delegate the PR-split to the operator upfront rather than folding.
- **Front-load `npm install` per package.** Every isolated package (`Spaarke.UI.Components`, `Spaarke.AI.Widgets`, `LegalWorkspace`, `SmartTodo`) needed `npm install --legacy-peer-deps` before local `tsc --noEmit` could run. Running them in parallel earlier (or as part of project scaffolding) would have saved ~2 minutes per package × 4 packages = 8 minutes.
- **Run `/test-diet` at Phase 1 checkpoint, not just at wrap-up.** The Phase 1 addition of 9 new tests in `DataGrid.recordOpen.test.ts` was clean per ADR-038, but a mid-project `/test-diet` sanity check would have caught any scaffolding drift earlier. Zero-cost given the report is short for TS-only projects.

## Technical debt intentionally accepted

- **`formDialogWidthPercent` / `formDialogHeightPercent` retained in schema but ignored at runtime.** Per R2 FR-20, all Layout 1 opens are 85% × 85%. Removing the deprecated fields entirely would be a schema-breaking cleanup (0 records use them per Task 001 audit, so low-risk); deferred as out-of-scope for R2. A future project could delete them.
- **`useLaunchContext.ts` `openTodo` discriminator is dead-in-practice.** No emitter in `src/` sends `?action=openTodo&todoId=<guid>` after Task 021's cleanup. Keeping the branch is ~15 lines of parser + type union that could be pruned. Not worth touching in R2 to avoid churn on unrelated tests.
- **`buildRecordOpenNavArgs` re-export decision is package-internal.** The helper is exported from `DataGrid.tsx` for the test file but intentionally NOT re-exported from the barrel `index.ts`. This matches the existing "package-internal test seam" pattern documented for `ColumnFilterType` and `ColumnFilterOption`. A `@internal` JSDoc tag on the helper would make the intent machine-readable for future TypeDoc/API-Extractor tooling.
- **No test coverage for `openSprkTodoAsLayout1` in Phase 3.** The helper is 20 lines and calls Xrm API directly. Testing it would require mocking `window.parent.Xrm`, which is complex for negligible payoff — observable behavior is fully covered by the manual QA checklist in `phase1-verification.md` (which extends to the To Do case). Not a KEEP-category miss per ADR-038.

## Follow-on work

- **Author per-entity "Workspace" simplified main form variants.** The `formId` capability (FR-01/FR-02) is shipped but unused; no `sprk_gridconfiguration` record sets `formId`. A future maker project could author simplified "Workspace" main form variants for Documents, Matters, Projects, etc., then set `formId` in the config records to have workspace row-clicks open the streamlined variant while form-embedded surfaces (subgrids, quick-create) continue to use the full main form.
- **Extract `LAYOUT_1_NAV_OPTIONS` to `@spaarke/ui-components`.** Both `DataGrid.tsx` and `SmartTodoApp.tsx` establish `{target:2, position:1, 85%, 85%}` independently. A shared const in `@spaarke/ui-components/constants/` would eliminate the duplication. Not urgent; the coupling seam ("DataGrid framework should not depend on `sprk_todo`-specific code and vice versa") makes duplication defensible.
- **Retire `useLaunchContext.ts` `openTodo` branch if truly unused.** Verify no ribbon/Outlook/external caller sends `?action=openTodo` after R2 deploys. If confirmed unused, prune the branch + type + tests. Out-of-scope for R2 to avoid coupling to launch-context callers under other projects.
- **Sub-savedqueries for Communications entity types.** The R2 Communications widget uses the OOB `Active Communications` savedquery which includes Email + Teams + SMS + Notification uniformly. If makers want type-specific views (Email-only, Notification-only), they can author sub-savedqueries and either point additional `sprk_gridconfiguration` records at them or override `configjson.source.savedQueryId`. Deferred as follow-up.
- **Consider component-level tests for `SmartTodoApp` mount/unmount.** Currently no test verifies that `SmartTodoApp` mounts without the removed `<SmartTodoModal>` — the manual QA covers observable behavior. If a component-mount test lands elsewhere in the codebase as a pattern, extend it here.

## What made this project close cleanly

- **Every phase had a discrete deliverable.** Phase 1 shipped framework + tests + baseline audit. Phase 2 shipped Communications end-to-end (record + section + direct widget + metadata). Phase 3 shipped retirement + 5 doc updates + CHANGELOG entry. No phase left partial state or "we'll fix it later" placeholders.
- **The R2 project's `CLAUDE.md` codified the two-layout binding standard early.** MUST rules like "MUST use 85% × 85% for every Layout 1 open (see FR-20)" made downstream decisions mechanical instead of debatable.
- **The researcher's evidence trail (`notes/researcher-iframe-main-aspx-2026-07-01.md`) meant the MS Learn 2025-05-07 quote was ready to inline at task 023.** Cite-once → reuse-many.
