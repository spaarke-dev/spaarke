## Overview

This is the R6 successor project. R6 left two parallel playbook-routing mechanisms, three parallel playbook-execution paths, hardcoded GUIDs / slash dicts / capability-name strings, and a stateless chat that re-asks users for already-uploaded files. This project collapses routing onto one matcher driven by `playbook-embeddings`, reforms playbook identification via stable `sprk_playbookcode` (already in schema; consumers must adopt), adds file-aware classification, builds a 6-tier stateful chat memory subsystem (mostly wire-and-refactor of existing R6 components), authors specialized playbooks (Summarize-NDA, Summarize-Patent, Extract-Invoice), and retires `CapabilityRouter` wholesale.

## Problem Statement

The chat-routing layer in R6 is functional but structurally fragile:

- **Two parallel routers**: `PlaybookDispatcher` (vector + LLM) and `CapabilityRouter` (Layer 0/0.5/1/2/3) — the latter was the layer R6 UAT hotfixes #1–4 had to defensively patch.
- **Hardcoded GUIDs and name lookups**: `SessionSummarizeOrchestrator.ChatSummarizePlaybookId`, `MatterPreFillService`, `ProjectPreFillService`, `WorkspaceFileEndpoints`, `WorkspaceAiService`, plus 4 name-resolve consumers in `AppOnlyAnalysisService`, `useAiSummary.ts`, `DocumentEmailWizard.tsx`, `ChatContextMappingService`. GUIDs don't transfer across environments; names collide silently.
- **Implicit-streaming Workspace destination**: today's Workspace tab opens via `sseToPaneEventBridge` reading `AnalysisChunk` deltas, NOT via data-driven `NodeRoutingConfig`. The streaming code is the implicit decider.
- **Stateless chat**: agent honestly tells users "I don't currently have the uploaded document's text fully in my context memory" turn-to-turn. Pillar 7 services + Cosmos containers exist but the layered prompt + tool surface + upload-time enrichment is not wired through.
- **Generic-only summarization**: `/summarize` invokes the generic SUM-CHAT@v1 for every document type. Uploading an NDA does not route to NDA-specific intelligence.

Users feel the symptoms in UAT (silent re-uploads, generic summaries, awkward turn taking). The audit pass showed the bugs are downstream consequences of the structural duplication, not isolated defects.

## Solution Summary

Six structural reforms — additive where possible, deletion where needed:

1. **One playbook-selection layer** — collapse `PlaybookDispatcher` + `CapabilityRouter` onto a single matcher driven by `playbook-embeddings`. The `sprk_aicapability` table is NOT built.
2. **Stable playbook codes** — `sprk_playbookcode` (already in schema; alternate key) becomes the canonical resolution path; migrate 9 hardcoded consumers via Pattern C cleanup → Pattern A (typed options + code) → Pattern B (name → code).
3. **Data-driven destinations** — wire `NodeRoutingConfig` (existing schema, correct shape) into `DispatchResult`; add `NodeDestination.Both`; add Workspace / Both / FormPrefill / SideEffect cases to `PlaybookOutputHandler`. Workspace tab opens via handler, not implicit streaming.
4. **Index governance** — 5 additive tracking fields on `sprk_analysisplaybook` + `sprk_jpsmatchingmetadata` JSON field; Power Apps "Send to Index" UX + nightly drift detection.
5. **6-tier stateful chat** — Working / Session / Matter / User-Org / Retrieval / Audit tiers with explicit promotion semantics. Mostly WIRE-and-REFACTOR of existing R6 Pillar 7 services. 8 new tool handlers; layered context cards; upload-time enrichment (classify + summarize + manifest); Harvey/Artifacts targeted-edit pattern for workspace tabs. Bound by `architecture/stateful-chat-architecture.md`.
6. **Specialized playbooks** — author `summarize-nda`, `summarize-patent`, `extract-invoice` using JPS `$ref` composition over existing Skills (SKL-003 NDA Review) + Knowledge (KNW-006 NDA Standards). Extend Path 3 (`ExecuteChatSummarizeAsync`) to resolve `$ref` while preserving streaming UX.

Then retire `CapabilityRouter` in a single-phase cutover (no parallel-run, per Q8).

## Scope

### In Scope

- §1.7 Playbook Identification Reform — stable codes consumer migration (Patterns A/B + cleanup C)
- WP1.5 Index Governance — Dataverse schema additions (5 fields) + Power Apps Send-to-Index UX + nightly drift detection
- WP2 File-aware classification — Hybrid (C) primary using precomputed upload manifests as pre-filter; (A) fallback on disagreement
- WP3 Destination metadata wiring (FR-14a through FR-14f — `Both` enum, `DispatchResult` extension, `PlaybookOutputHandler` cases, deploy-time validation)
- WP4 CapabilityRouter retirement — single-phase cutover; no backward compat
- WP5 6-tier stateful chat — bound by `architecture/stateful-chat-architecture.md` §3–§4–§8
- WP6 Specialized playbooks + Path 3 JPS `$ref` extension (additive only per audit correction)
- ADR-030 v2 `memory` channel — already amended in this branch (2026-06-21)

### Out of Scope

- Modification of the 6 production-bound playbooks per spec §1.5 (migrate in place via stable code; no delete; no rename; no output-schema change)
- `voyage-law-2` embeddings migration (defer; 6–10% precision loss accepted)
- Azure Document Intelligence classifier (wrong tool for Stage 1 routing)
- Foundry-pattern long-term memory (Q4 — defer)
- `sprk_matterfacts` separate Dataverse entity (use Cosmos `memory`)
- Detailed promote-to-matter-memory UX wireframes (architecture §6.4 flow agreed; visual design deferred)
- `sprk_userpreferences` field expansion (Q13 defer; chat reads existing shape via cached snapshot)
- BFF microservice extraction for chat memory (rejected per ADR-013; latency-coupled with playbook routing)
- Insights Engine internal reuse — `spaarke-insights-index`, `MultiIndexComposer`, `InsightsOrchestrator`, `EvidenceSufficiencyNode`, `GroundingVerifyNode`, `sprk_performancesummary` (binding boundary per architecture §5)
