## Problem Statement

The BFF's distributed-cache layer drifted: dev BFF runs with `Redis__Enabled=false` because the underlying Redis instance was deleted at some prior point, and the App Setting was left at `false` ever since. The in-memory fallback was active in a deployed environment without anyone noticing. ADR-009 says "Redis-first" but provides no operational MUSTs for SKU, fail-fast, secret management, tenant key isolation, or observability — making this drift undetectable. The current `CacheModule` swallows the connection failure (`AbortOnConnectFail = false`), and the in-memory `IDistributedCache` it falls back to does not propagate cache invalidations across multi-instance App Service deployments. Cache keys today carry no tenant prefix, which is an invariant violation for a multi-tenant platform. The legacy resource (`spe-redis-dev-67e2xz`) name is also off-pattern for Spaarke's canonical naming convention.

## Solution Summary

Architecture 1 (one platform Redis per environment) with mandatory tenant key prefix and a wrapper API (`ITenantCache`) that is future-extensible for multi-Redis. Phase 1 hardens `CacheModule` (fail-fast, env-guarded `AllowInMemoryFallback`, Null-Object `IConnectionMultiplexer` per ADR-032), introduces the wrapper, and atomically migrates **all** direct `IDistributedCache` call sites in `Sprk.Bff.Api`. Phase 2 extends `redis.bicep` + extracts `Deploy-RedisCache.ps1` from `Provision-Customer.ps1`. Phase 3 provisions `spaarke-bff-redis-dev` (Basic C0) and cuts the dev BFF over to it via Key Vault reference. Phases 4–5 wire App Insights observability and finalize canonical docs + ADR-009 amendments (concise + full, in lockstep).

## Key Decisions

| Decision | Rationale | ADR |
|----------|-----------|-----|
| Architecture 1 (Platform-only Redis) | One Redis per environment + tenant key prefix; future-extensible wrapper for multi-Redis without redesign | ADR-009 (amended) |
| Wrapper as new interface (`ITenantCache`) | ≥2 future implementations (default today + future named instances per NFR-12); satisfies ADR-010 + CLAUDE.md §11 three-question test | [ADR-010](../../.claude/adr/ADR-010-di-minimalism.md) |
| Null-Object `IConnectionMultiplexer` | Symmetric DI registration prevents asymmetric anti-pattern (P3 fail-fast tier) | [ADR-032](../../.claude/adr/ADR-032-bff-nullobject-kill-switch.md) |
| KV reference for connection string | No plain-text secrets in App Settings; MI reads on App Service | [ADR-028](../../.claude/adr/ADR-028-spaarke-auth-architecture.md) |
| Atomic 117-(actually ~199)-call-site migration | Prevents read-old/write-new bug class across multi-tenant key space | spec NFR-07 |
| Industry-standard key format `{InstanceName}tenant:{tenantId}:{resource}:{id}:v{version}` | Drops deprecated `sdap` brand; multi-tenant invariant | spec FR-05 / Q-A |
| 24-hr verification window (not 48) | Legacy dev Redis is empty — no data to verify | spec FR-15 / Q3 |
