# Smart To Do — Decoupling from Events (R3)

> **Last Updated**: 2026-06-07
>
> **Status**: Draft

## Overview

R3 replaces the event-coupled `sprk_event + sprk_eventtodo` to-do model with a first-class `sprk_todo` custom entity that stands alone or attaches to any of eleven parent record types via Spaarke's multi-entity resolution pattern. To-dos mirror bidirectionally to Microsoft Graph `/me/todo`, surfacing in Outlook, Teams Tasks, and the Microsoft To Do app. The Outlook add-in gains a "Create To Do" ribbon action and an indicator for emails with linked Spaarke to-dos.

## Problem Statement

The current `sprk_event` + `sprk_eventtodo` design from events-smart-todo-kanban-r1/r2 forces every to-do to attach to an event, even when the user's intent is a standalone task or a task regarding a Matter, Project, Communication, etc. The two-entity model also blocks consolidated kanban data access (parallel loads of `sprk_event` and `sprk_eventtodo` per board) and prevents native integration with Microsoft Graph `/me/todo`, which is the on-ramp to Outlook, Teams Tasks, and the Microsoft To Do app. The result is a Spaarke-only experience that does not meet users where they already manage tasks.

## Solution Summary

Introduce a first-class `sprk_todo` custom entity with the full kanban behavior + multi-entity resolution attribute set, retire `sprk_eventtodo` entirely (pre-release, no shims), hoist `KanbanBoard` / `KanbanColumn` / `KanbanCard` to `@spaarke/ui-components` so SmartTodo and LegalWorkspace share one implementation, mirror records bidirectionally to Microsoft Graph `/me/todo` per opted-in user (own list per user, with deep-link back via Modern UCI), and update the Outlook add-in with a "Create To Do" ribbon action + linked-todo indicator. Loop prevention combines `sprk_synchash` + thread-local skip-flag + per-field last-write-wins.

## Scope

### In Scope

- New `sprk_todo` custom entity (kanban + Graph sync state + 11 specific regarding lookups + 4 resolver fields)
- Retirement of `sprk_eventtodo` and removal of `sprk_todoflag` / `sprk_todostatus` / `sprk_todocolumn` / `sprk_todopinned` from `sprk_event`
- Kanban hoist from SmartTodo-local to `@spaarke/ui-components`
- `TodoDetail` shared component simplified to single-entity load/save
- SmartTodo Code Page repointed at `sprk_todo`; "My Tasks" filter added
- CreateTodo wizard repointed at `sprk_todo`; `AssociateToStep` integrated for regarding
- "To Dos" subgrid added to each of the eleven parent entity main forms
- Microsoft Graph `/me/todo` bidirectional sync (per-user opt-in, backfill, last-write-wins, Modern UCI deep link)
- New `Tasks.ReadWrite` delegated scope + reuse of existing OBO infrastructure
- Outlook add-in ribbon action "Create To Do" (email-read context) + indicator banner

### Out of Scope

- Backward-compatibility shims for `sprk_eventtodo` or `sprk_event.sprk_todo*` (hard cut, pre-release)
- Data migration of existing dev-environment `sprk_event` (`todoflag=true`) or `sprk_eventtodo` rows
- Microsoft Planner / group-owned task integration
- Legacy `/me/outlook/tasks` API (deprecated by Microsoft Aug 2022)
- Custom Teams app, personal tab, or bot extension
- Changes to the scoring formula, threshold defaults, drag/pin behavior, or general kanban UX
- Editing existing to-dos from the Outlook add-in (v1 is create-only + indicator)
- Mobile-native Spaarke surface (mobile UX delivered indirectly via MS To Do mobile app)

## Key Decisions

| Decision | Rationale | ADR |
|----------|-----------|-----|
| Custom entity (not Activity) for `sprk_todo` | Spaarke doesn't use Activities; kanban + scoring need full customization | — |
| Multi-entity resolution (not polymorphic `regardingobjectid`) | Spaarke standard; reuses `PolymorphicResolverService` + `AssociateToStep` | [ADR-024](../../.claude/adr/ADR-024-polymorphic-resolver-pattern.md) |
| Hard cut (no compat shims) | Pre-release; future maintenance burden of shims outweighs migration value | — |
| Last-write-wins per-field (timestamp-based) | Matches user mental model; cleaner than entity-level wins | — |
| Modern UCI deep link scheme | Long-term Microsoft direction; preserves form context | — |
| Per-user opt-in for Graph sync | Privacy + delegated-token cost; respects user choice of task surface | — |
| Single `User` lookup for `sprk_assignedto` | Teams own via standard `ownerid`; team-assignment out of scope | — |
| Graph sync feature-gated (Null-Object pattern) | BFF starts cleanly without Graph creds in dev / when sync disabled | [ADR-032](../../.claude/adr/ADR-032-bff-nullobject-kill-switch.md) |

## Risks & Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Graph 429 throttling under initial backfill | Med | Med | Exponential backoff (1→2→4→8→cap 30s) + circuit breaker (5 consecutive 429s → 5-min pause) per NFR-05 |
| Subscription expiry not renewed → silent sync loss | High | Low | Nightly renewal job within 24h of expiry; failure alarmed (NFR-07) |
| BFF publish-size delta breaches 60 MB ceiling | High | Low | Per-task publish-size verification per CLAUDE.md §10 / `bff-extensions.md` |
| Webhook `clientState` leak compromises sync auth | High | Low | Store in Key Vault with rotation pipeline (UQ-3 resolution) |
| Bidirectional sync loop (echo of own writes) | Med | Med | `sprk_synchash` + thread-local skip-flag + per-field LWW (FR-23) |
| Initial backfill long-running, interruptible | Low | Med | Resumable with `initialBackfillCompletedUtc` checkpoint; UI toast shows progress (FR-20) |
| Stale Modern UCI app id per environment | Med | Med | `DeepLinkBuilder` reads from config (`Spaarke:ModelDrivenApps:DefaultAppId`); environment-owner confirmation (UQ-2) |
