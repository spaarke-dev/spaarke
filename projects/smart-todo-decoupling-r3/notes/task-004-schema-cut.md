## Surprises / lessons

1. **Hidden Copilot blocker**: Power Platform's "Form Fill" Copilot feature auto-creates `aiskillconfig` records for every field that a user has opted out of. These are stored as ComponentType=10314 (`aiskillconfig`) in `solutioncomponents` and they **block attribute deletes** with a generic "referenced by 1 other component" error. The `RetrieveDependenciesForDelete` response identifies them only by GUID + numeric componenttype — translating that to the actual entity name (`aiskillconfigs`) required experimentation. This blocker class is likely to recur on every future field deletion in this environment. Suggest filing this as a `script-aware` recipe for future schema cuts.
2. **`RemoveSolutionComponent` API ergonomics**: the OData ActionImport advertises `SolutionComponent` (entity ref) as the parameter name in `$metadata`, but the server-side handler legacy code requires `ComponentId`. The two are mutually exclusive — you cannot satisfy both. The path that ultimately worked here was sidestepping `RemoveSolutionComponent` entirely and DELETEing the underlying `aiskillconfig` row directly. Other component types may require different surface APIs.
3. **`mcp__dataverse__describe` is the cleanest post-cut verification surface** — it returns a T-SQL-style CREATE TABLE that makes it instantly obvious which columns are present/absent, with no need to parse OData responses.
